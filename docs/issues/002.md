## 前提

- Dockerfile にあるとおりの Python バージョン
- `main_001_discord_to_jsonl.jsonl` にあるような形式の jsonl を扱う

## 作りたいスクリプトの仕様

main_002_rawjsonl_to_parsedjsonl.py

1. コマンドライン引数として以下をとる
    - `--input`: 必須
    - `--output`: 任意 (デフォルト: `main_002_rawjsonl_to_parsedjsonl.jsonl`)
    - `--limit`: 任意 (デフォルト: None)
    - `--parser`: 任意 (デフォルト: "default")
2. 以下の "パーサー仕様" に従って、1行ずつ jsonl をパースする
3. 以下の "検証仕様" に従ってパース結果を検証する
4. パース結果を jsonl 形式で保存する
    - ディレクトリ: output/{引数で指定されたもの}

### パーサー仕様

ひとつの content の中の、以下のような形式の文字列だけを扱う。ほかの行は切り捨てる。  
つまり markdown のリスト形式。  
よって `-` も `*` も `+` も扱う。

```
- foo
    - bar
    - baz
- qux
    - quux
```

以下のような json へ変換。

```json
[
    {
        "heading": "foo",
        "children": [
            "bar",
            "baz"
        ]
    },
    {
        "heading": "qux",
        "children": [
            "quux"
        ]
    }
]
```

### 検証仕様

#### ひとつめ

たまにこういうデータがある。 (このプログラムがおかしいんじゃなくて、もともとの discord 側のメッセージがそうなってるだけ)

```json
[
]
```

まあつまり、パース結果が空っぽってやつ。空っぽのレコードがあるよ、と警告メッセージを出す。
もともとのメッセージの、最初の20文字くらいを表示して "..." とする、 created_at を JST で表示、ほかは不要。

#### ふたつめ

たまにこういうデータがある。 (このプログラムがおかしいんじゃなくて、もともとの discord 側のメッセージがそうなってるだけ)

```json
[
    {
        "heading": "foo",
        "children": []
    }
]
```

こういうのがあったら discord 側のメッセージを直したいので、 heading はあるけど children のないメッセージがあるよ、と警告メッセージを出す。
メッセージの heading を表示、 created_at を JST で表示、ほかは不要。
